## 一、决策汇总

经过 4 轮需求讨论，确认以下设计决策：

| 编号 | 议题 | 决策 |
|------|------|------|
| Q1 | 数据库选型 | **MySQL** |
| Q2 | 状态机设计 | 无"接单"状态，assigned 即进行中；工程师回复"已解决"同步给用户；用户反馈"未解决"直接给工程师 |
| Q3 | current_load 计算 | **动态查询**，不再手动维护 JSON 字段 |
| Q4 | easy 任务未解决处理 | 走 assign_node 匹配工程师；无匹配则**默认分配第一位工程师** |
| Q5 | 工程师"已解决"是否终态 | **方案 A：直接 resolved 终态**，通知用户 |
| Q6 | 反馈消息与任务关联 | **方案 A：追踪最近任务**，按钉钉会话 ID 关联用户最近一条 active 任务 |
| Q7 | 负载均衡策略 | **混合策略**：LLM 筛技能（返回候选人列表）+ 算法做负载均衡 |
| Q8 | 同负载平局处理 | **随机选择**，优先分配没有任务的工程师（load=0） |
| Q9 | 匹配工程师全部不可用 | **方案 B**：仍只看技能匹配，从 available=false 中选负载最低的，通知后等待 |
| Q10 | LLM 返回格式 | 返回候选人列表 `{"candidates": [...], "reason": "..."}`，0 个则默认第一位 |
| Q11 | ORM 选型 | **SQLAlchemy** |
| Q12 | 工程师名单存储 | **迁移到 DB engineers 表**，废弃 engineers.json |
| Q13 | "已解决"识别方式 | **关键词匹配** |
| Q14 | 表结构 | 确认 tasks / engineers / feedbacks 三表设计 |

## 二、状态机设计

### 2.1 核心状态（3 个）

```
┌────────────────┐     用户反馈"未解决"      ┌──────────────┐     工程师回复"已解决"     ┌──────────┐
│ auto_answered  │ ─────────────────────────→ │   assigned   │ ───────────────────────→ │ resolved │
│  (自动已回答)   │                             │  (已分配)     │                           │  (已解决)  │
└────────────────┘                             └──────────────┘                           └──────────┘
        │                                           │                                          ▲
        │ 用户反馈"已解决"                            │ 用户反馈"未解决"                           │
        └───────────────────────────────────────────┼────────── 重新催办工程师 ──────────────────┘
                                                    ↓ 保持 assigned
```

### 2.2 状态流转规则

| 当前状态 | 触发条件 | 目标状态 | 动作 |
|---------|---------|---------|------|
| （新任务） | classify → easy + answer 完成 | `auto_answered` | 存库，等待用户反馈 |
| （新任务） | classify → hard + assign 完成 | `assigned` | 存库，通知工程师 |
| `auto_answered` | 用户反馈"已解决" | `resolved` | 更新状态，记录反馈 |
| `auto_answered` | 用户反馈"未解决" | `assigned` | 走 assign_node 分配工程师，通知工程师 |
| `assigned` | 工程师回复"已解决" | `resolved` | 更新状态，通知用户"已处理" |
| `assigned` | 用户反馈"未解决" | `assigned`（不变） | 重新催办对应工程师 |
| `assigned` | 用户反馈"已解决" | `resolved` | 用户确认解决，关闭任务 |

### 2.3 说明

- **无 `pending` / `classified` / `in_progress` 中间态**：任务入库时直接是终态之一（`auto_answered` 或 `assigned`）
- **`resolved` 是唯一终态**：归档后不再流转
- **工程师回复"已解决"即终态**（方案 A），无需用户二次确认

### 3.2 建表 SQL

```sql
-- 工程师表（从 engineers.json 迁移）
CREATE TABLE engineers (
    id              INT AUTO_INCREMENT PRIMARY KEY,
    name            VARCHAR(100) NOT NULL UNIQUE COMMENT '姓名',
    skills          JSON NOT NULL COMMENT '技能标签列表',
    mobile          VARCHAR(20) DEFAULT '' COMMENT '手机号',
    dingtalk_user_id VARCHAR(64) DEFAULT '' COMMENT '钉钉UserID',
    available       BOOLEAN DEFAULT TRUE COMMENT '是否在岗',
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at      DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='IT工程师名单';

-- 任务表
CREATE TABLE tasks (
    id                INT AUTO_INCREMENT PRIMARY KEY,
    task_no           VARCHAR(20) NOT NULL UNIQUE COMMENT '任务编号 T1001',
    title             VARCHAR(500) NOT NULL COMMENT '标题',
    description       TEXT NOT NULL COMMENT '描述',
    submitted_by      VARCHAR(100) NOT NULL COMMENT '提交人姓名',
    submitter_id      VARCHAR(64) DEFAULT '' COMMENT '提交人钉钉ID(反馈追踪用)',
    difficulty        ENUM('easy','hard') NOT NULL COMMENT '难度',
    status            ENUM('auto_answered','assigned','resolved') NOT NULL COMMENT '任务状态',
    assigned_engineer VARCHAR(100) DEFAULT '' COMMENT '分配工程师姓名',
    final_response    TEXT COMMENT '给用户的回答/分配信息',
    created_at        DATETIME DEFAULT CURRENT_TIMESTAMP,
    resolved_at       DATETIME DEFAULT NULL COMMENT '解决时间',
    INDEX idx_submitter (submitter_id, status),
    INDEX idx_engineer (assigned_engineer, status)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='运维任务';

-- 反馈记录表
CREATE TABLE feedbacks (
    id            INT AUTO_INCREMENT PRIMARY KEY,
    task_id       INT NOT NULL COMMENT '关联任务ID',
    feedback_type ENUM('resolved','unresolved') NOT NULL COMMENT '反馈类型',
    feedback_by   VARCHAR(100) NOT NULL COMMENT '反馈人',
    created_at    DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (task_id) REFERENCES tasks(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='任务反馈记录';
```

## 四、消息路由逻辑（核心）

这是反馈闭环的关键：钉钉收到消息后，如何判断是"新任务"还是"反馈"。

### 4.1 整体路由流程图

```
收到钉钉消息
    │
    ▼
识别发送者身份（查 engineers 表）
    │
    ├───【是工程师】
    │        │
    │        ▼
    │   查该工程师是否有 status='assigned' 的任务？
    │        │
    │        ├── 有 → 检查是否命中"已解决"关键词
    │        │        ├── 命中 → 标记 resolved，通知提交人 ✅
    │        │        └── 未命中 → 当作补充信息，记录日志，不改状态
    │        │
    │        └── 无 → 当作新任务提交（走 classify_node）
    │
    └───【是普通用户】
             │
             ▼
        查该用户是否有 active 任务？
        (submitter_id 匹配, status IN ('auto_answered','assigned'))
             │
             ├── 有 active 任务 → 检查反馈意图
             │        │
             │        ├── "已解决"类关键词 → 标记 resolved ✅
             │        │
             │        ├── "未解决"类关键词 →
             │        │     ├── 当前 auto_answered → 走 assign_node 分配工程师 → assigned
             │        │     └── 当前 assigned → 重新催办工程师（状态不变）
             │        │
             │        └── 其他 → 当作追问（第一阶段：重新走完整 graph 回答）
             │
             └── 无 active 任务 → 当作新任务，走 classify_node
```

## 五、负载均衡算法

### 5.1 算法流程

```
assign_node(task)
    │
    ▼
① LLM 分析任务，返回候选人列表
   Prompt 输出: {"candidates": ["张三","李四"], "reason": "需要网络技能"}
    │
    ▼
② 从 engineers 表筛选候选人（校验 LLM 返回的名字真实存在）
    │
    ├── 候选人为空（LLM 返回 0 个或名字都不存在）
    │       → 默认分配第一位工程师
    │
    ▼
③ 在候选人中，优先选 available=true 的
    │
    ├── 有在岗的 → pool = 在岗候选人
    └── 全部不在岗（方案B）→ pool = 不在岗候选人
    │
    ▼
④ 动态计算 pool 中每人的 current_load
   (SELECT COUNT(*) FROM tasks WHERE assigned_engineer=? AND status='assigned')
    │
    ▼
⑤ 选 current_load 最低的
   finalists = [e for e in pool if e.load == min_load]
    │
    ▼
⑥ 优先选 load=0 的工程师（没有任务优先分配）
   zero_load = [e for e in finalists if e.load == 0]
    │
    ├── 有 zero_load → 从中随机选 1 个
    └── 无 zero_load → 从 finalists 中随机选 1 个
    │
    ▼
⑦ 返回选中的工程师 + 原因，发通知
```

### 5.2 算法伪代码

```python
import random

def assign_engineer(task) -> tuple[Engineer, str]:
    """
    混合负载均衡分配算法。
    返回 (工程师对象, 分配原因)
    """
    all_engineers = load_engineers_from_db()

    # Step 1: LLM 返回候选人
    candidates_names = llm_match_candidates(task, all_engineers)

    # Step 2: 校验候选人是否真实存在
    matched = [e for e in all_engineers if e.name in candidates_names]

    if not matched:
        # 无候选人 → 默认第一位
        return all_engineers[0], "无匹配工程师，默认分配"

    # Step 3: 优先在岗，全部不在岗则用不在岗的（方案B）
    available_pool = [e for e in matched if e.available]
    pool = available_pool if available_pool else matched

    # Step 4: 动态计算负载
    for e in pool:
        e.current_load = count_active_tasks(e.name)

    # Step 5: 选最低负载
    min_load = min(e.current_load for e in pool)
    finalists = [e for e in pool if e.current_load == min_load]

    # Step 6: 优先 0 负载，同负载随机
    zero_load = [e for e in finalists if e.current_load == 0]
    chosen = random.choice(zero_load if zero_load else finalists)

    reason = f"技能匹配，当前任务数最少（{chosen.current_load}个）"
    if not chosen.available:
        reason += "（注意：该工程师当前不在岗，可能响应较慢）"

    return chosen, reason
```

## 六、反馈闭环详细设计

### 6.1 用户反馈"已解决"（auto_answered → resolved）

```
用户在钉钉回复"搞定了" 
    ↓
路由判断：该用户有 active 任务，状态 auto_answered
    ↓
命中 USER_RESOLVED_KEYWORDS
    ↓
更新任务 status = resolved, resolved_at = now()
    ↓
记录 feedbacks 表 (type=resolved)
    ↓
回复用户："好的，已为您关闭任务 #T1001，感谢反馈！"
```

### 6.2 用户反馈"未解决"（auto_answered → assigned）

```
用户在钉钉回复"还是不行"
    ↓
路由判断：该用户有 active 任务，状态 auto_answered
    ↓
命中 USER_UNRESOLVED_KEYWORDS
    ↓
更新任务 status = assigned
    ↓
记录 feedbacks 表 (type=unresolved)
    ↓
调用 assign_engineer() 分配工程师（走负载均衡算法）
    ↓
更新 assigned_engineer 字段
    ↓
通知工程师（钉钉私聊 + 群简报）
    ↓
回复用户："自动回答未能解决问题，已为您转给工程师 {name}，请稍候。"
```

### 6.3 工程师回复"已解决"（assigned → resolved）

```
工程师在钉钉回复"已解决"
    ↓
路由判断：该工程师有 active 任务，状态 assigned
    ↓
命中 ENGINEER_RESOLVED_KEYWORDS
    ↓
更新任务 status = resolved, resolved_at = now()
    ↓
记录 feedbacks 表 (type=resolved, feedback_by=工程师名)
    ↓
通知提交人（钉钉私聊）："您的任务 #T1001 已由工程师 {name} 处理完成，如仍有问题请回复。"
```

### 6.4 用户催办（assigned 状态，用户反馈未解决）

```
用户在钉钉回复"还没解决"
    ↓
路由判断：该用户有 active 任务，状态 assigned
    ↓
命中 USER_UNRESOLVED_KEYWORDS
    ↓
状态不变（仍 assigned）
    ↓
记录 feedbacks 表 (type=unresolved)
    ↓
重新通知工程师（钉钉私聊）："用户反馈任务 #T1001 仍未解决，请尽快跟进。"
    ↓
回复用户："已再次通知工程师 {name}，请稍候。"
```

### 8.2 反馈处理流程（graph 外，feedback.py 处理）

```
钉钉消息
    │
    ▼
identify_sender()
    │
    ├── 工程师 + 有active任务 + "已解决"关键词
    │       → update resolved → 通知用户
    │
    ├── 用户 + 有active任务 + "已解决"关键词
    │       → update resolved
    │
    ├── 用户 + active=auto_answered + "未解决"关键词
    │       → update assigned → assign_engineer() → 通知工程师
    │
    ├── 用户 + active=assigned + "未解决"关键词
    │       → 重新催办工程师（状态不变）
    │
    └── 其他（新任务 / 追问）
            → 走主 graph (classify → ...)
```

> **注意**：反馈处理不经过 LangGraph，直接在 `feedback.py` 中操作数据库 + 调用 `assign_engineer()`（复用负载均衡逻辑）。这样主 graph 保持简洁，反馈逻辑独立可测。

## 九、边界情况处理

| 场景 | 处理方式 |
|------|---------|
| 工程师名单为空（DB 无数据） | assign_engineer 返回错误，通知"暂无可用工程师，请联系 IT 主管" |
| LLM 返回候选人都不在 engineers 表中 | 降级为默认第一位工程师 |
| 用户同时有多个 active 任务 | 方案A只追踪最近一个，反馈作用于最近任务 |
| 工程师回复非"已解决"的普通消息 | 记录日志，不改状态，不回复（或简单回复"收到"） |
| 用户追问（非已解决/未解决关键词） | 第一阶段：当作新任务重新走 graph（简化处理） |
| 数据库连接失败 | 降级：任务不入库，仍走原 graph 流程回答/通知，日志告警 |
| API 调用（非钉钉）创建的任务 | submitter_id 为空，不参与反馈闭环（仅钉钉渠道支持反馈） |
| engineers.json 迁移时表已有数据 | 跳过迁移，以 DB 数据为准 |
| 任务编号生成并发冲突 | 用 DB 自增 ID 拼接，保证唯一：`T + id补零至4位` |

## 十、数据迁移方案

### 10.1 engineers.json → engineers 表

在 `main.py` 启动时执行一次性迁移：

```python
def migrate_engineers_json_to_db():
    """首次启动：将 engineers.json 导入数据库"""
    json_path = DATA_DIR / "engineers.json"

    # 表已有数据，跳过
    if db_manager.count_engineers() > 0:
        print("[迁移] engineers 表已有数据，跳过 JSON 迁移")
        return

    if not json_path.exists():
        print("[迁移] engineers.json 不存在，跳过")
        return

    with open(json_path, "r", encoding="utf-8") as f:
        engineers = json.load(f)

    for e in engineers:
        db_manager.create_engineer(e)

    print(f"[迁移] 已导入 {len(engineers)} 位工程师到数据库")
```
